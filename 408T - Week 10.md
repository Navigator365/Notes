# Lab 1

Using procmon, we see that our program tries to delete temp1\lab1.txt. We can trick the program into deleting lab1\lab1.txt by creating a junction at temp1. Annoyingly, we can't make an already-existing directory a junction, so we must first delete temp1, then recreate it as a junction to lab1 via `mklink /J temp1 lab1`
# Lab 2

Using procmon, we see that our program tries to delete temp2\temp.txt. We'll create an Object directory and create a temp.txt file with a symlink to lab2\lab2.txt, which we're allowed to do, as even unpriviledged users can Object Manager symlinks. Then, we'll create temp2 as a junction to our Object directory, so that accessing temp2\temp.txt first points us to our Object directory, and then to lab2.txt through that directory's symlink. 

We'll use symlink-testing-tools to do all this work for us through `` CreateSymlink.exe temp2\temp.txt lab2\lab2.txt``
# Lab 3

We can create a register symlink to the key we want to delete, which will in turn delete its value. Through procmon, we can see our program deletes HKCU\SOFTWARE\koolkevin\Lab3Value, so we can run symlink-testing-tools `CreateRegSymlink.exe HKCU\SOFTWARE\koolkevin HKCU\SOFTWARE\lab3`to create the symlink, which will then be used to delete the Lab3Value when our program terminates. 

# Lab 4

In this lab, our program checks that temp4\temp.txt is a normal file, and then deletes it. We can use a TOCTOU exploit to get around this, setting a delete-based oplock on the file. In this way, the program can read the file and see that it's normal, but can't delete it until we release the oplock. When we do so, we'll immediately delete `temp4\*` and create an object symlink from temp4\temp.txt to lab4\lab.txt. We'll maintain this symlink long enough for the program to delete temp4\temp.txt, which will delete lab4\lab.txt. 

We'll do this programmatically. We'll use the same basic structure as SetOptLock.cpp in symlink-testing-tools. We'll change a few arguments around: we're only optlocking on delete, so we can hardcode that. Then, right after calling `delete oplock`, we'll recursively delete the `temp4` directory (this can be done with many code snippets found online, I used [this one](https://blog.j2i.net/2025/05/27/recursively-deleting-directory-in-win32-c/)), then use FileOpLock.h's CreateSymlink, hardcoding the path as we do so. We'll include a getc statement to ensure our program doesn't return (and delete the symlink) before we want it to. 

Note that this might not always work, since it's a race condition, but we'll win most of the time.

# Lab 5

We can see the homework binary creates a directory named 0000-9999, and a file temp.txt in it. Then, it creates a log.txt file, and finally deletes the temp.txt file. We can perform a TOCTOU attack similar to Lab3. 

First, we'll write-oplock on log.txt. We oplock on write to allow the homework binary to read log.txt and confirm it's a normal file. Once the oplock is triggered, we know the temporary directory and file have just been created, so we can search through our current directory to find the most-recently modified 4-digit directory, which will be the homework's directory. 

Once we've found this directory, we'll delete it and replace tempdir\temp.txt with a symlink to lab5\lab5.txt. Then, once our oplock is released, the homework binary will delete the file following the symlink, and delete lab5\lab5.txt. 

Finally, I'll note that this race condition was relatively sensitive; I had to hold the oplock until I had already found and deleted the tempory directory/file.
